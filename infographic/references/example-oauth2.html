What is OAuth 2.0?

Authorization framework — NOT authentication

OAuth 2.0 lets a user grant limited access to their resources on one service, to another service — without sharing their password. Think: "Allow Spotify to read your Facebook friends list."

❌ Without OAuth

"Give me your Google password
so I can read your contacts"

DANGEROUS — Full access, no revoke

VS
✅ With OAuth

"Google, give this app a token
to read ONLY my contacts"

SAFE — Scoped, time-limited, revocable

💡 Key insight: OAuth separates "who you are" from "what you can access." It's a valet key — not the master key.

The 4 Roles

Every OAuth flow has exactly these four actors
👤
Resource Owner

The user who owns the data.
"Yes, Spotify can see my friends."

📱
Client

The app requesting access.
Spotify, Notion, Slack, etc.

🏛️
Authorization Server

Issues tokens after consent.
Google's auth endpoint.

📦
Resource Server

Holds the protected data.
Google's Contacts API.

💡 Remember: Auth Server and Resource Server are often the same company (Google) but logically separate. The Auth Server handles consent; the Resource Server validates tokens.

THE MAIN FLOW
▼
03
🔄

Authorization Code Flow

The most secure & most common grant type — used by 80% of apps
1. User clicks "Sign in with Google"

Client redirects to Google's /authorize endpoint with client_id + scope + redirect_uri

2. User sees consent screen

"Spotify wants to: View your contacts, Read your profile" — User clicks Allow

3. Google redirects back with auth code

redirect_uri?code=abc123 — This code is SHORT-LIVED (60 seconds)

4. Client exchanges code for tokens

POST /token with code + client_secret → receives access_token + refresh_token

5. Client calls API with access token

GET /contacts with Authorization: Bearer <token> → gets protected data

// Step 4: Exchange code for token POST /oauth/token { "grant_type": "authorization_code", "code": "abc123", "client_id": "spotify-app", "client_secret": "s3cret", "redirect_uri": "https://spotify.com/callback" }

💡 Why the extra step? The auth code travels through the browser (visible). The token exchange happens server-to-server (hidden). This keeps the access_token safe from browser-based attacks.

Token Lifecycle

Two tokens, different purposes
🔓 Access Token

Lifetime: 15 min – 1 hour
Format: JWT or opaque
Purpose: API authorization
Sent via: Authorization header

+
🔄 Refresh Token

Lifetime: Days to months
Format: Always opaque
Purpose: Get new access tokens
Stored: Server-side only

💡 Why short-lived access tokens? If stolen, damage is limited to minutes. The refresh token (stored securely server-side) gets new access tokens silently. Users never re-login.

Scopes — Granular Permissions

The "what" of authorization — not all-or-nothing
// Request only what you need (principle of least privilege) scope="read:contacts" // ✅ Read contacts only scope="read:contacts write:calendar" // ✅ Contacts + calendar scope="admin" // ❌ Too broad — users will deny
✅ Good Scopes

read:profile
read:email
write:posts

⚠️ Broad Scopes

user:all
admin:full
offline_access

❌ Over-Scoping

"Give me everything"
Users won't trust you
Violates least privilege

OAUTH vs OIDC
▼
07
🆔

OAuth 2.0 vs OpenID Connect

OAuth = authorization · OIDC = authentication (built on top of OAuth)
OAuth 2.0

Answers: "What can this app do?"
Token: access_token
Use: API access, data sharing
Example: "Let Slack read my calendar"

+
OpenID Connect

Answers: "Who is this user?"
Token: id_token (JWT)
Use: Login, identity verification
Example: "Sign in with Google"

💡 Simple rule: Need to access an API? Use OAuth. Need to log a user in? Use OIDC. Most apps use both together.

Security Best Practices

Common attacks and how to prevent them
🎭
CSRF Attack

Attacker forges auth callback
Fix: Use state parameter

🔗
Token Leakage

Token exposed in URL/logs
Fix: Use PKCE, not Implicit

🔄
Token Replay

Stolen token reused
Fix: Short expiry + rotation

🎯
Redirect Attack

Malicious redirect_uri
Fix: Exact URI matching
